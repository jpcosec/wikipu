**Abstract:** The Socratic Protocol is the structured interrogation workflow that identifies hidden assumptions, missing constraints, and architectural contradictions before a plan is authorized.

The Socratic Protocol is the structured interrogation workflow executed by an agent (or human) before a plan or proposal is authorized for implementation. It systematically identifies hidden assumptions, missing constraints, and architectural contradictions to guarantee ID-1 Orthogonality and ID-2 Minimal Energy.

## Rule Schema

The protocol executes a sequence of analytical passes over a proposed change:

1. **Intention Parsing:** Extract the core goal and structural claims of the proposal.
2. **Graph Validation:** Query the Knowledge Graph to verify if the intent overlaps with existing nodes.
3. **Constraint Identification:** Expose unstated assumptions or missing system constraints required to execute the proposal safely.
4. **Edge Case Probing:** Identify scenarios where the proposed logic would violate a House Rule.
5. **Artifact Generation:** Output a typed list of questions or required revisions to `desk/socratic/Board.md`.

No proposal may be moved to `desk/tasks/` until all questions generated by this protocol have been explicitly resolved and encoded into the plan.

## Fields

A Socratic evaluation produces findings categorized by the following types:
- `missing_constraint`: A required rule or boundary is not defined.
- `unstated_assumption`: The plan relies on a state that is not guaranteed.
- `contradiction`: The plan conflicts with an existing node or House Rule.
- `undefined_edge_case`: A failure mode is unhandled.
- `scope_creep`: The plan violates OP-5 Atomization and needs splitting.
- `ownership_gap`: It is unclear which domain owns the proposed change.

## Usage Examples

An agent acting under the Socratic Protocol reviews a proposal to add a new database module:

**Agent Action:**
1. Parses the proposal.
2. Queries the graph: `wiki-compiler query --type get_node --node-id "concept:storage"`.
3. Identifies an overlap with existing data-layer constraints.
4. Outputs a `contradiction` finding: "The proposal introduces a direct database connection, which violates the existing contract pattern defined in `contracts.py`. How should this be abstracted?"

❯ what are drafts?                                                              

● wiki/drafts/ is the output directory for wiki-compiler ingest. When you run:

  wiki-compiler ingest --source raw/ --dest wiki/drafts/

  It scans every .md file in raw/, decomposes each one into atomic sections, and
   writes a draft node per section into wiki/drafts/. Each draft gets a YAML
  frontmatter header with a node_id, node_type, and an edge back to its source
  file.

  So the 1530 files in wiki/drafts/ are the compiled output of ingesting the 6
  external project doc trees (raw/docs_cotizador/, raw/docs_doc_methodology/,
  etc.). They're machine-generated stubs — not curated wiki nodes. The intention
   is that a human (or agent) would review them, promote the useful ones into
  real wiki nodes, and discard the rest.

  In the current state: none of them have been reviewed or promoted. They're
  sitting there as unprocessed output.

  The distinction in the system design is:
  - wiki/ = curated, canonical nodes written for this repo
  - wiki/drafts/ = raw ingest output, staging area, not authoritative

  So yes, they're ephemeral artifacts — the intermediate product of the ingest
  step, not the finished wiki.

  ---
  query-server-runtime

  A / B / C:

  B (Python API) + A (CLI wrapper). Not C.

  The Librarian agent calls bash commands, so CLI is the interface. But the CLI
  should be a thin wrapper over a Python API — the real implementation lives in
  query_executor.py as callable functions. A long-running server adds
  operational complexity (start/stop, health checks, port ownership) with no
  clear benefit: loading knowledge_graph.json per query is fast enough at this
  scale. If it gets slow later, in-process caching is the right fix, not a
  daemon.

  What should submit_topology_proposal return beyond CollisionReport:

  The current CollisionReport tells you whether there's a collision and which 
  nodes collide. What's missing is what to do about it. I'd add:

  - suggested_alternatives: list[str] — node IDs the agent should extend/reuse
  instead of creating a new node. If your proposal collides with
  code:src/wiki_compiler/scanner.py:scan_files, the report should say "reuse
  this instead."
  - overlap_reasons: list[str] — human-readable explanation per collision (which
   IOFacet fields overlapped, which semantic terms matched).
  - auto_approvable: bool — true when there are zero collisions. Lets the agent
  short-circuit without waiting for human approval.

  FacetProposal support:

  Yes, include both. TopologyProposal adds nodes; FacetProposal enriches
  existing ones. They're different validations but the same interface pattern.
  FacetProposal is actually simpler (check orthogonality only within the target
  node's existing facets), so it's lower-effort to add and agents will need it
  constantly.

  ---
  cleansing-protocol

  Input for --apply:

  B — full report with approved: true/false per proposal.

  Workflow: cleanse --detect > report.json → human edits the file, sets
  approved: true on the ones they want → cleanse --apply report.json. This
  creates a clear audit trail (you can see what was rejected, not just what was
  applied) and handles the full set in one pass. Option A (one proposal at a
  time) loses the context of the full detection pass.

  What should --apply mutate:

  Files on disk first, then rebuild the graph. Never graph-only. The graph is
  derived from files — a graph-only change gets overwritten on the next build.
  Files are truth. The sequence: mutate frontmatter / delete files / rename
  nodes → trigger build_wiki() internally → updated graph reflects the changes.

  Destructive actions require explicit per-proposal approval:

  Yes. Any proposal with requires_human_approval: true that doesn't have
  approved: true explicitly set is skipped, not applied. --apply should log
  which proposals were skipped and why. No implicit approvals, no "apply all"
  shortcut for destructive actions.

● I'll take them one at a time.

  ---
  1 — Git-tracked node IDs

  The instinct is right but the implementation needs care. Git tracks content as
   blob SHAs and paths as tree entries. If you embed the blob SHA in the node ID
   itself (file:src/builder.py@a3f4c2d), every commit that touches the file
  changes its node ID and breaks all edges pointing to it. That's worse than the
   current situation.

  Better split: keep the path-based ID stable as the permanent identifier, add
  git metadata as a facet. A GitFacet per node with: created_at_commit,
  last_modified_commit, blob_sha_at_build_time, author. This lets you answer
  "when was this node introduced?", "who last touched it?", "what changed
  between graph builds?" without breaking the edge topology. The blob SHA is a
  snapshot for drift detection — if it changes between builds, the node is
  stale.

  The deeper value: edges get a first-class lifecycle. You can ask "was this
  depends_on edge valid at commit X?" which is currently impossible.

  ---
  2 — Socratic Protocol

  The Socratic method works by surfacing hidden assumptions through structured
  questioning — not to find answers, but to find what you don't know you're 
  assuming. What we just did with the query-server and cleansing questions is
  exactly this: I listed the assumptions embedded in each issue and asked you to
   resolve them before implementation starts.

  For wikipu this should be a named, invocable protocol — not just ad hoc Q&A.
  The protocol would:
  - Take a plan, design doc, or proposed node as input
  - Emit a set of open questions organized by type: missing constraints,
  unstated assumptions, contradictions with existing nodes, undefined edge cases
  - Store the Q&A as a structured artifact (not just conversation)

  It's prospective where Phase 6 is retrospective. Phase 6 patches rules after
  friction. Socratic runs before work starts, to prevent the friction. The two
  are complementary — Socratic is the pre-flight check, Phase 6 is the
  post-flight debrief.

  ---
  3 — Trail collect

  Every conversation produces artifacts that currently evaporate: design
  decisions, gap discoveries, corrections, Q&A resolutions, things the agent got
   wrong and had to fix. The methodology synthesis found the same principle as
  Phase 6 — "no logical error should survive the session that discovered it" —
  but that requires a mechanism.

  Trail collect is that mechanism. At the end of a session (or at natural
  breakpoints), a step classifies conversation artifacts by type and routes
  them:
  - Design decision resolved → encode in the relevant doc or hausordnung
  - Gap discovered → create an issue
  - Q&A that resolved an ambiguity → save to raw/Q&A/ or inline into the
  relevant doc
  - Correction (agent was wrong) → update the source of the wrong assumption

  The output is not a transcript — it's structured extractions. The raw session
  log stays in Claude's memory system; trail collect distills only the durable
  facts. This is the operationalization of Phase 6.

  ---
  4 — Autopoiesis / self-fixing system

  Autopoiesis (Maturana & Varela): a system is autopoietic when it continuously
  produces the components that constitute it — the system's operation produces
  the conditions for its own continuation. A cell synthesizes the proteins that
  make the cell. The system and its production process are the same thing.

  Applied here: the wiki should produce its own repair components. When the
  graph detects a stale node, it generates the proposal to fix it. When a gap is
   found, it generates the issue. When a rule causes friction, it generates the
  patch to the hausordnung. The system's outputs are also its inputs.

  The autoresearch analogy is about directed self-inquiry — the system sets its
  own research agenda. Combined: the system detects what it doesn't know,
  formulates questions, runs the investigation, and integrates the results back
  into itself.

  The loop for wikipu would be: build → detect anomalies (cleanser) → surface
  questions (Socratic) → generate proposals → human approves → apply → trail
  collect → rebuild → repeat. Each iteration, the system is more complete and
  more coherent than the last. This is exactly the "self-correction loop" from
  the compact note — but framed as a designed, reentrant process rather than a
  one-off session habit.

  The constraint that separates this from fully autonomous self-modification:
  human approval is the gate for destructive or structural changes. The system
  can diagnose and propose; it cannot unilaterally restructure itself. This is
  the same constraint as cleanse --apply requiring explicit per-proposal
  approval.

  └─────────────────────┴───────────────────────────────────────┘

  resolve on its own. One central monitoring point means a human checks one file
   (Index.md) instead of five different inboxes.

  needs a name that captures "things the system is waiting on a human for." A

  - Desk — informal, "what's on the human's desk right now"

  between proposal and execution), and it's replicable: every subsystem that
  needs HITL has a Gate. The issues system is then one instantiation of a Gate.
  plan_docs/issues/Index.md is the issues Gate. A future
  plan_docs/proposals/Index.md would be the proposals Gate.

  Autopoiesis in biological systems can be viewed as a network of constraints   
  that work to maintain themselves. This concept has been called organizational 
   closure[7] or constraint closure[8] and is closely related to the study of   
  autocatalytic chemical networks where constraints are reactions required to   
  sustain life.                                                                 
                                                                                
  Moreover, an autopoietic system is autonomous and operationally closed, in    
  the sense that there are sufficient processes within it to maintain the       
  whole. Autopoietic systems are "structurally coupled" with their medium,      
  embedded in a dynamic of changes that can be recalled as sensory-motor        
  coupling.[10] This continuous dynamic is considered as a rudimentary form of  
  knowledge or cognition and can be observed throughout life-forms.             
                                                                                
  In the context of textual studies, Jerome McGann argues that texts are        
  "autopoietic mechanisms operating as self-generating feedback systems that    
  cannot be separated from those who manipulate and use them".[16] Citing       
  Maturana and Varela, he defines an autopoietic system as "a closed            
  topological space that 'continuously generates and specifies its own          
  organization through its operation as a system of production of its own       
  components, and does this in an endless turnover of components'", concluding  
  that "Autopoietic systems are thus distinguished from allopoietic systems,    
  which are Cartesian and which 'have as the product of their functioning       
  something different from themselves'". Coding and markup appear allopoietic", 
   McGann argues, but are generative parts of the system they serve to          
  maintain, and thus language and print or electronic technology are            
  autopoietic systems.[17]: 200–1

● Good question. Let me work through the gaps systematically.

  For true autopoiesis you need three things the system currently lacks:

  ---
  1. Perception — the system can't sense its own perturbations

  Right now every process requires human invocation. build, ingest, cleanse —
  all manually triggered. An autopoietic system responds to perturbations; this
  one waits to be told. It needs the ability to detect: a file appeared in raw/,
   src/ changed since last build, a node's blob_sha drifted, a gate has been
  open for N days. Without perception, there's no autonomous response — just a
  collection of tools.

  ---
  2. Self-inclusion — the system doesn't model itself

  The graph contains nodes for code, docs, wiki. But the CLI commands, the agent
   protocols, the identity rules themselves — these are not nodes. The system
  maintains a knowledge graph about everything except its own processes. This
  violates operational closure: a system can't maintain itself if it doesn't
  know its own components. The hausordnung, the processes, the gate mechanism —
  all need to be nodes in the graph they govern.

  ---
  3. The identity layer is informal and unenforced

  The identity rules (orthogonality, minimal energy, typed contracts, zone
  separation) exist as text in raw/. They are not machine-readable, not enforced
   by any process, and not checked on any build. 00_house_rules.md doesn't even
  exist yet. Without formalized, enforced identity rules, the system has no
  invariant — it's just a collection of files that could drift into anything.

  ---
  4. The response suite is incomplete

  The autopoietic cycle is: detect perturbation → classify → generate response →
   gate if structural → apply → collect trail → rebuild. Currently:

  ┌─────────────────┬───────────────────────────────────────────────────────┐
  │      Step       │                        Status                         │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Detect          │ Missing (no perception)                               │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Classify        │ Partial (audit exists, no continuous classification)  │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Generate        │ Partial (CleansingProposal planned, TopologyProposal  │
  │ response        │ exists)                                               │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Gate            │ Missing (desk/Gates.md doesn't exist)                 │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Apply           │ Partial (cleanse --apply planned)                     │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Collect trail   │ Missing                                               │
  ├─────────────────┼───────────────────────────────────────────────────────┤
  │ Rebuild         │ Exists                                                │
  └─────────────────┴───────────────────────────────────────────────────────┘

  ---
  5. No cross-session memory

  Each session starts cold. Decisions made in session N are not available to
  session N+1 unless explicitly encoded somewhere. The autopoietic loop requires
   continuity — the system needs to remember what perturbations it has already
  responded to, what anomalies were found and resolved, what gates are open.
  Without this, each session re-derives from scratch.

  ---
  The minimal energy rule needs to be active, not declared

  It currently exists as a concept. For it to function as an identity
  constraint, every process that creates or modifies elements must evaluate: is 
  there a simpler structure that satisfies this requirement? This needs to be a
  check in the topology validation step, not just a principle in a document.

  ---
  Summary — the three missing layers:

  Identity layer     →  machine-readable, enforced hausordnung + self-inclusion
  in graph
  Perception layer   →  drift detection, change watchers, perturbation
  classification
  Response layer     →  cleanse, curate, trail collect, socratic, gate
  mechanism, loop coordinator

  Of these, identity layer first — you can't enforce autopoietic closure without
   knowing what the invariants are. The hausordnung is the precondition for
  everything else. Without it, any process you build will optimize for the wrong
   thing.

# Data Management

Schema-v0 runtime contract for pipeline artifacts.

---

## Purpose

This document defines where generated data lives, who owns it, and how the
pipeline interacts with it.

The code remains the source of truth for domain contracts:

- Pydantic schemas in `contracts.py`
- CLI signatures in `_build_parser()` inside each `main.py`

This document only defines the runtime data-plane rules.

---

## Non-Negotiable Rules

1. Every artifact generated by a pipeline step must be persisted under `data/`.
2. LangGraph owns control-plane state and checkpoints, not artifact payload
   storage.
3. Job roots are physically immutable once created.
4. Urgent, archived, and discarded are metadata states, not alternate folders.
5. The top-level graph may pass lightweight data in memory between steps, but a
   step is not complete until its canonical outputs are written to disk.
6. `src/core/data_manager.py` only reads and writes primitive data types:
   `dict`, `str`, `bytes`, and metadata.
7. Nodes and adapters validate primitive payloads into Pydantic models; the data
   manager never imports feature-domain contracts.

---

## Canonical Layout

All schema-v0 jobs live under:

```text
data/jobs/<source>/<job_id>/
```

Each job root contains a single lifecycle file:

```text
meta.json
```

And per-node outputs under:

```text
nodes/<node_name>/<stage>/<artifact>
```

Example:

```text
data/jobs/stepstone/12345/
```

## Lifecycle Metadata

`meta.json` is managed centrally by the data manager.

Initial schema-v0 fields:

- `schema_version`
- `status` (`active`, `urgent`, `archived`, `discarded`)
- `created_at`
- `updated_at`

The pipeline may read this metadata for filtering and operational decisions, but
it must not move job roots on disk.

---

## Ownership By Layer

### LangGraph state

Owns:

- routing
- UI-facing lightweight summaries
- refs and status flags

Must not own:

- full generated documents
- large source markdown
- repeated JSON payload copies for every step

### Data manager

Owns:

- job root resolution
- node/stage artifact placement
- primitive artifact reads and writes
- lifecycle metadata in `meta.json`

Enforcement rule:

- runtime code under `src/graph/`, `src/core/ai/`, and `src/core/tools/` must not bypass `DataManager` for filesystem operations
- direct `Path.read_text()`, `read_bytes()`, `write_text()`, `write_bytes()`, and `mkdir()` calls are only allowed inside `src/core/data_manager.py`

Must not own:

- domain validation
- business logic
- LangGraph state

### Nodes and adapters

Own:

- validating primitive data into Pydantic models
- business transforms
- deciding what the canonical outputs of the step are

---

## Current Migration Rule

Schema-v0 may redesign outputs freely.

Step inputs should stay stable unless changing them clearly improves the runtime
contract enough to justify the migration cost.

---

## Standalone Commands vs Graph Runtime

### Top-level graph

- reads through the data manager when loading disk artifacts
- works on in-memory primitive payloads and validated models
- persists canonical outputs back through the data manager

### Standalone CLI

Follows a strict three-step flow:

1. `DataManager.read_*_artifact(...)`
2. pass primitive payload into the domain adapter/core
3. `DataManager.write_*_artifact(...)`

This keeps disk IO out of business logic while preserving debuggable standalone
commands.
